# Knowledge Graph — Validation

## Purpose

`knowledge_graph/validation.py` provides three categories
of validation for KG data:

1. **Temporal consistency** — enforced at save time.
2. **Alias collision detection** — advisory audit.
3. **Relationship constraint checking** — advisory audit.


## Temporal consistency

### What it checks

`valid_until >= valid_from` on both `Entity` and
`Relationship` objects. Called automatically from
`save_entity()`, `save_relationship()`, and
`save_relationships()` before the database INSERT.

### When it raises

Only when **both** `valid_from` and `valid_until` are
non-None and `valid_until < valid_from`. All other
combinations are valid:

| valid_from | valid_until | Result |
|------------|-------------|--------|
| None       | None        | OK — unbounded |
| set        | None        | OK — open end |
| None       | set         | OK — open start |
| set        | set (>=)    | OK — bounded range |
| set        | set (<)     | **ValidationError** |

### Why save-time, not `__post_init__`?

Dataclasses are frozen and used extensively in tests with
arbitrary values. Adding `__post_init__` validation would
break many test fixtures and make it harder to construct
test objects. Save-time validation catches issues when
data enters the store, which is the boundary that matters.


## Alias collision detection

### What it checks

`find_alias_collisions(conn)` queries the `entity_aliases`
table for aliases shared by two or more entities (case-
insensitive). Returns `AliasCollision` objects with the
conflicting alias and the entities that share it.

### Why advisory?

The LLM may legitimately produce entities with overlapping
aliases:

- A ROLE entity "CEO" and a PERSON entity nicknamed "CEO"
- Two organizations in different countries with the same
  abbreviation
- Aliases that are contextually unambiguous but textually
  identical

Blocking on save would reject valid data. Instead, call
`find_alias_collisions()` periodically or after bulk
ingestion to flag potential conflicts for human review.

### Usage

```python
collisions = find_alias_collisions(store._conn)
for c in collisions:
    print(f"Alias '{c.alias}' shared by:")
    for eid, name, etype in c.entities:
        print(f"  - {name} ({etype})")
```


## Relationship constraint checking

### What it checks

`check_relationship_constraints()` validates a single
relationship against the canonical patterns documented in
`docs/knowledge_graph/relationships.md`. Returns a list of
warning strings (empty if the pattern is known).

`audit_relationship_constraints(conn)` scans all
relationships in the KG and reports those that don't match
any documented pattern.

### Constraint table

The `RELATIONSHIP_CONSTRAINTS` dict maps
`(source_type, target_type)` pairs to frozensets of known
`relation_type` strings. Patterns are derived from
`relationships.md` and cover:

- Market structure (asset issuance, listing, tracking)
- People and roles (employment, leadership, founding)
- Analyst coverage
- Products (manufacturing, regulation)
- Sector events
- Regulation and policy
- Economic indicators
- Corporate actions
- Credit ratings
- Competitive and supply chain
- Corporate structure
- Location and membership
- Sector and topic linkage

### Why advisory?

Relationship types are free-form strings generated by the
LLM. The documented patterns are conventions, not
exhaustive rules. Blocking unknown patterns would reject
valid novel relationships that the LLM discovers in
articles. Instead, audit periodically to flag unusual
patterns for review.

### Usage

```python
# Single relationship check
warnings = check_relationship_constraints(
    "works_at",
    EntityType.PERSON,
    EntityType.ORGANIZATION,
)
# warnings == [] (known pattern)

# Full KG audit
warnings = audit_relationship_constraints(store._conn)
for w in warnings:
    print(f"{w.source_type} -> {w.target_type}: "
          f"{w.relation_type} — {w.message}")
```


## What was deferred

- **Subtype validation** — checking that subtypes match
  canonical values from `subtypes.md`. Subtypes are
  free-form strings by design, so validation would be
  advisory only and low-priority.
- **Cross-entity temporal consistency** — e.g., a
  relationship's temporal bounds should fall within the
  temporal bounds of both participating entities. Complex
  to define and enforce, deferred until needed.
- **Circular relationship detection** — e.g., A -> B and
  B -> A with the same relation_type. Not necessarily
  wrong (competition is symmetric), so needs careful
  definition before implementation.
